## The Pull Model

Traditional CI/CD pushes: a pipeline builds an image, then `kubectl apply`s it to the cluster. GitOps inverts this — the cluster _pulls_ its desired state from Git. ArgoCD watches a repo, compares it to what's running, and reconciles the difference. Git is the source of truth; the cluster is a follower.

## Why Pull

Push pipelines need credentials to talk to the cluster, and they need to know _which_ cluster. Pull needs neither — the cluster authenticates to Git, not the other way around. That means your CI never holds cluster credentials, and the same repo can drive dev, staging, and prod with no pipeline changes.

## ArgoCD Reconciliation

```bash
$ argocd app list
NAME       CLUSTER   STATUS     HEALTH   SYNC
api        https://k8s  Synced    Healthy  Auto
worker     https://k8s  OutOfSync Degraded Manual
```

ArgoCD reports drift — when someone hand-edits a resource in the cluster, the app goes `OutOfSync`. The fix is a commit, not a `kubectl` command. This is the core promise: every change is auditable, because every change is a commit.

## Sync Waves

When resources depend on each other (a CRD before its instance, a namespace before its workloads), sync waves order them. Annotate a resource with `argocd.argoproj.io/sync-wave: "-1"` and it applies before wave 0.

## Conclusion

GitOps isn't a tool — it's the discipline of making Git the only way to change cluster state. ArgoCD is the reconciler that enforces it. Once you trust the pull loop, deploys stop being a pipeline you babysit and become a commit you review.
