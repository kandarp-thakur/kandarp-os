## What State Is

Terraform state is the map between your config and the real resources it created. Without it, Terraform can't tell what it owns versus what already existed. Lose the state, and Terraform thinks your entire infrastructure is missing — and cheerfully offers to rebuild it.

## Remote State

Never store state locally. A remote backend (S3 + DynamoDB, or Terraform Cloud) gives you a shared, durable, lockable state file. The team shares one source of truth; the lock prevents two people applying at once.

```hcl
terraform {
  backend "s3" {
    bucket         = "my-tf-state"
    key            = "prod/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "tf-locks"
    encrypt        = true
  }
}
```

## Locking

The DynamoDB table is the lock. `terraform apply` writes a lock item; a second apply blocks until the first releases it. Without locking, two concurrent applies corrupt the state — one overwrites the other's changes.

## The Force-Push Incident

A teammate once `git push --force`'d over a branch that had hand-edited state. The next `apply` tried to destroy half the VPC. The lesson: state lives in the backend, not in Git, and `force` is never safe on a branch anyone else depends on.

## Workspaces

Workspaces let one config manage multiple environments (dev, staging, prod) with isolated state files. They're convenient, but they share the same backend — make sure the state keys are namespaced so environments can't clobber each other.

## Conclusion

State is the most fragile part of Terraform. Treat it like production data: remote, locked, backed up, and never hand-edited. The day you lose it is the day you understand why.
