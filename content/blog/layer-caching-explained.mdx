## The Cache Problem

A Docker image is a stack of layers — one per instruction in your `Dockerfile`. Rebuilding an image re-runs every instruction unless its inputs haven't changed, in which case Docker reuses the cached layer. The whole performance game is ordering instructions so the cache hits as often as possible.

## How Layers Work

Each `RUN`, `COPY`, and `ADD` creates a layer. Layers are content-addressed: the cache key is a hash of the instruction _and_ the files it touches. Change a file, and every layer from that point down is rebuilt.

```dockerfile
FROM node:20
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci
COPY . .
RUN npm run build
```

## Cache Keys

The trick is to copy only what `npm ci` needs (`package.json` + lockfile) _before_ copying the rest of the source. That way, editing `src/index.ts` doesn't invalidate the `npm ci` layer — your dependencies stay cached across source changes.

## Multi-Stage Builds

A full build image carries the compiler, the source, and the toolchain — hundreds of megabytes you don't need at runtime. Multi-stage builds let you compile in one stage and copy only the artifact into a slim final image.

```dockerfile
# build stage
FROM node:20 AS build
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# final stage
FROM node:20-slim
WORKDIR /app
COPY --from=build /app/dist ./dist
COPY package*.json ./
RUN npm ci --omit=dev
CMD ["node", "dist/index.js"]
```

## Benchmarking

On a typical Node service, reordering the `COPY` cuts a no-cache rebuild from ~90s to ~12s on a cache hit. Multi-stage then trims the final image from ~1.1GB to ~180MB.

## Summary

Cache is about ordering: copy the lockfile first, install deps, then copy source. Multi-stage is about size: build fat, ship thin. Together they turn a slow, bloated build into a fast, slim one.
