## History

Control groups (cgroups) were merged into the Linux kernel in 2.6.24 (2008). They let you group processes and apply resource limits — CPU, memory, I/O — to the group as a whole. Every container runtime (Docker, containerd, runc) is, at its core, a cgroups + namespaces frontend.

## v1 vs v2

cgroups v1 had one hierarchy per resource controller. You could limit CPU in one tree and memory in another, with no relationship between them — a process could be in different cgroups for different controllers, which made accounting ambiguous.

cgroups v2 (merged in 4.5, matured through 5.x) introduced a **unified hierarchy**: a single tree where every controller attaches to the same cgroup. A process lives in exactly one cgroup, and all controllers see it there.

## Unified Hierarchy

```bash
$ mount | grep cgroup
cgroup2 on /sys/fs/cgroup type cgroup2 (rw,nosuid,nodev,noexec,relatime)
$ ls /sys/fs/cgroup
cgroup.controllers  cgroup.subtree_control  cpu.stat  ...
```

The unified hierarchy means a process's resource usage is accounted once, consistently, across all controllers. No more "which cgroup is this process really in?" ambiguity.

## Controllers

The v2 controllers are a subset of v1, deliberately simplified:

- `cpu` + `cpu.weight` — proportional CPU sharing
- `memory.max` — hard memory limit
- `io.max` — I/O bandwidth limits
- `pids.max` — process count limit

## Practical Examples

Limit a shell to 512MB of memory and 200 PIDs:

```bash
$ mkdir /sys/fs/cgroup/myshell
$ echo 512M > /sys/fs/cgroup/myshell/memory.max
$ echo 200 > /sys/fs/cgroup/myshell/pids.max
$ echo $$ > /sys/fs/cgroup/myshell/cgroup.procs
```

Now any process you launch from that shell is capped at 512MB and 200 PIDs — a poor-man's container, in three commands.

## Migration

Most modern distos ship with cgroups v2 enabled by default (systemd 244+). Container runtimes support it via `--cgroupns` and the `cgroup2` driver. The migration is largely transparent; the main gotcha is tooling that still reads v1 paths under `/sys/fs/cgroup/cpu/...`.

## Conclusion

cgroups v2 is the quiet foundation under every container. Understanding the unified hierarchy makes "why did my container get OOM-killed?" a question you can actually answer — the limit is right there in `memory.max`.
